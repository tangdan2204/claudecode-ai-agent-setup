## 二、已执行的优化（本次审查后立即修复）

### P0-1: hooks.json 不存在的脚本引用（已修复）
- **问题**: hooks.json SessionStart 引用了 `hud-plus-guard.sh`，但项目中不存在此文件
- **修复**: 添加存在性检测，不存在时优雅降级 `test -f ... && ... || true`

### P0-2: settings.json deny 覆盖不足（已修复）
- **问题**: 原有 16 条规则未覆盖 sudo、git push --force、eval、curl|bash 等关键威胁
- **修复**: 新增 8 条 deny 规则，覆盖 sudo/eval/git push --force/git reset --hard/curl|bash/wget|bash
- **效果**: deny 规则从 16 条增至 24 条，消除了 safety-guard.sh 的单点依赖

### P1-1: 熔断机制从警告升级为硬阻止（已修复）
- **问题**: post-edit-audit.sh 同文件编辑 8 次仅输出警告（exit 0），AI 可以完全忽略
- **修复**: ≥8 次时改为 `exit 2` 硬阻止，将御史台殿院的"建言权"升级为"弹劾权"
- **效果**: 熔断不再依赖 AI 的"自觉性"，形成真正的制衡

### P1-2: 敏感信息检测模式扩展（已修复）
- **问题**: 缺少 SSH/TLS 私钥、JWT Token、Azure 连接字符串检测
- **修复**: 新增 4 种模式，敏感信息检测从 15 种增至 19 种

### P1-3: verify-before-stop.sh JSONL 兼容（已修复）
- **问题**: 审计日志已升级为 JSONL 格式，但解析仍用旧的 `awk -F'|'`
- **修复**: 改为 `jq -r '.file'` 解析 JSONL

## 三、架构层次评估

### 3.1 优秀之处
- **拦截与辅助分离清晰**: 3 层硬拦截（exit 2）+ 4 层辅助监控，职责明确
- **生命周期覆盖完整**: 7 个 hook 覆盖 SessionStart/PreToolUse/PostToolUse/PreCompact/Stop/Notification
- **纵深冗余正确**: settings.json deny 与 safety-guard.sh 有意重叠，非冗余而是纵深

### 3.2 待改进之处
- **"8 层纵深防御"语义膨胀**: 只有 3 层是真正的安全拦截，建议描述更精确
- **pre-compact-save/restore 配对缺乏容错**: 写入失败时恢复流程静默降级
- **正则存在绕过路径**: Base64 编码、环境变量替换、xargs 间接执行

## 四、三省六部执行体系评估

### 4.1 历史映射评价
- **三省核心映射准确**: 中书省→决策/planner、门下省→审核/verifier、尚书省→执行/executor
- **六部映射精妙**: 吏部→Agent调度（铨选=Agent选拔）、工部→构建部署 最为精确
- **偏差之处**: 兵部→安全防御（历史上兵部是军事管理非防御）、刑部→审计追踪（刑部掌刑狱非追踪）

### 4.2 制衡有效性
| 封驳类型 | 实现方式 | 强制力 | 评价 |
|----------|----------|--------|------|
| 硬封驳 | Hook exit 2 | 绝对强制 | 不可绕过，设计优秀 |
| 软封驳 | Reviewer 意见 | 仅为建议 | 缺乏强制执行机制 |
| 条件封驳 | 熔断计数 | **已升级为硬阻止** | 本次修复后从 exit 0 改为 exit 2 |

### 4.3 建议引入的古制元素
- **考课制 → Agent 绩效评估**: 追踪每类 Agent 的成功率，动态调整路由权重
- **制敕文书制度 → 结构化审批记录**: 在 `.omc/state/decree-log.jsonl` 记录三省流转
- **谏官角色**: 对中等以上复杂度任务强制运行 critic agent

## 五、开发原则审查详细发现

### 5.1 KISS 原则（最需关注，4/10）
- CLAUDE.md 395 行 + OMC 注入 300 行 = ~700 行行为指令，超出 AI 可靠遵循的上限
- 七阶段认知循环可简化为四阶段：理解(感知+思考) → 规划 → 执行+验证 → 学习(反思+进化)
- 三省六部框架占 35 行但未引入新行为规则（所有映射指向已有组件），可降级为设计参考文档

### 5.2 OCP 原则（5/10）
- 安全规则硬编码在脚本内，无法不修改脚本即扩展
- **建议**: 将规则抽取到 `rules/dangerous-commands.txt` 和 `rules/sensitive-patterns.txt`

### 5.3 防御编程（8/10）
- `set -euo pipefail` 全覆盖
- `jq 2>/dev/null` 错误处理全覆盖
- 日志轮转（>500 行截断到 200 行）
- **风险**: 并发写入 `edit-audit.log` 可能导致行交错，建议使用 flock

### 5.4 自主进化（6/10）
- `recurring-patterns.md` 只记录了 1 个模式，统计仪表板显示全局审计触发 0 次
- 进化机制完全依赖 AI "自觉执行"，没有 hook 强制检查
- Agent 采访协议设计精妙但实际上无法回溯性采访已完成的 agent

## 八、审查结论

> 该项目在"提示词工程 + Shell Hook"这个约束空间内，已经做到了接近最优的设计。
> 硬安全层（Hook exit 2）坚固可靠，但软治理层（prompt 约束）的制衡机制缺乏执行保障——
> 恰如唐代中后期门下省封驳权被架空的历史教训。
> 本次审查已将条件封驳从"建言权"升级为"弹劾权"（exit 2），部分弥补了这一缺陷。

**审查完成。三个 Agent 审查报告交叉验证，共发现 23 个问题点，已当场修复 5 个 P0/P1 级别问题。**
